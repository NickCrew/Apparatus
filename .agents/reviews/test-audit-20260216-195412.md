**Current test count:** 2 tests
**Estimated coverage:** ~15% (manual audit)

---

## Summary

The SSE broadcast module is **severely under-tested**. The existing test file tests the *middleware integration* (that requests trigger broadcasts and that `/sse` is excluded), but it does **not test the `SSEBroadcaster` class itself** — the core of the module. None of the class's public methods (`addClient`, `broadcast`, `getClientCount`, `getMaxClients`), its DoS protection (max client limit), heartbeat logic, client cleanup, SSE wire format, or any of the four convenience functions are directly tested. The module manages shared mutable state (a client Map) and handles connection lifecycle events, making it a reliability-critical component that needs substantially more coverage.

- ✅ Covered: 2

- ❌ Missing: 22

## Shallow Test Details

Neither existing test is technically "shallow" — they both make meaningful assertions on specific behavior. However, they have **scope limitations** worth noting:

### Middleware integration test (`should broadcast request details after response finishes`)
**Current test:** Makes a `GET /healthz` request via supertest, verifies `broadcast` was called with matching method/path/status.
**Observation:** This tests the *middleware* that calls `broadcastRequest()`, not the broadcaster itself. The spy on `broadcast` means the actual SSE write path is never exercised. This is correct for a middleware test but does not substitute for unit tests of the broadcaster.
**No fix needed** — but the gap it leaves in broadcaster coverage must be filled by new unit tests.

### SSE exclusion test (`should NOT broadcast requests to /sse endpoint`)
**Current test:** Opens `/sse`, aborts the connection, verifies no broadcast occurred.
**Observation:** Uses `req.abort()` and a 100ms sleep — somewhat fragile timing-wise, but the assertion is meaningful. The mock of `sseHandler` prevents a hanging test, which is pragmatic.
**No fix needed** — consider replacing the `setTimeout` with a more deterministic approach if this test becomes flaky.

## Notes

### Test Infrastructure Gaps

1. **No mock Response helper exists.** Nearly every unit test for `SSEBroadcaster` needs a mock Express `Response` object with `write()`, `writableEnded`, and EventEmitter capabilities (`once`, `emit`). Create a `createMockSSEResponse()` test helper that returns a mock Response with:
   - `write` as a `vi.fn()`
   - `writableEnded` as a settable property (default `false`)
   - EventEmitter behavior for `close`, `error`, `finish` events (extend `EventEmitter` or use `vi.fn()` for `once`)

2. **Singleton contamination.** `sseBroadcaster` is a module-level singleton. Tests that add clients will contaminate subsequent tests unless cleanup is thorough. The existing `beforeEach` calls `removeAllListeners()` but does NOT clear the `clients` Map (it's private). Options:
   - Add a `reset()` or `removeAllClients()` method for testing
   - Or: each test should trigger cleanup by emitting `close` on all mock responses
   - Or: use `vi.importActual` / dynamic imports to get fresh instances

3. **Fake timers needed.** Heartbeat testing requires `vi.useFakeTimers()` to avoid 30-second waits. Timestamp assertions also benefit from frozen time.

4. **The `clients` Map is private.** Direct assertions on internal state aren't possible without either:
   - Relying on `getClientCount()` as the observable proxy (preferred)
   - Adding a test-only accessor (avoid if possible)
