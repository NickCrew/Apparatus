## Notes

### Test Infrastructure Requirements

1. **Mock `useSSE`:** The hook depends on `useSSE` which manages the actual SSE connection. Tests should mock `useSSE` to control `status` and `subscribe` behavior. This is an appropriate boundary mock — SSE is an external I/O boundary.

2. **Mock `useApparatus`:** The hook pulls `baseUrl` from `ApparatusProvider` context. Tests should either wrap in a test provider or mock the hook.

3. **Use `@testing-library/react` with `renderHook`:** Standard approach for testing React hooks in isolation. Allows controlled rerenders and unmount simulation.

4. **Simulating event streams:** The `subscribe` mock should accept a callback and allow tests to invoke it manually with synthetic `TrafficEvent` payloads, simulating incoming SSE data.

### Suggested Test Utility

A helper like this would reduce boilerplate across all test cases:

```typescript
function setupTrafficStream(options?: {
  bufferSize?: number;
  initialStatus?: string;
  baseUrl?: string;
}) { ... }
```

This would wire up the mocks for `useApparatus` and `useSSE`, return the `renderHook` result plus a `pushEvent(event)` function and a `setStatus(status)` function for driving the test.

### Observation on `bufferSize = 0`

The current implementation handles `bufferSize = 0` by accepting the event into state and then immediately truncating to an empty array on the next check (`updated.length > 0` is true, `slice(0, 0)` returns `[]`). This means every event triggers a state update that results in `[]` — wasted renders. This may warrant either a guard clause (`if (bufferSize <= 0) return;`) or explicit documentation that 0 means "discard all events." A test should capture whichever behavior is intended.

### Observation on Stale Closure Risk

The `setEvents` callback uses the functional updater form (`prev => ...`), which correctly avoids stale closure over `events`. This is good. However, `bufferSize` is captured in the closure and is listed in the effect's dependency array, so changes to `bufferSize` correctly tear down and re-subscribe. A test confirming the functional updater behavior (multiple rapid events don't drop intermediate state) would add confidence.
